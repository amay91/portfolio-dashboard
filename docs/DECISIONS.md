# Decisions & bug‑fix log

Why the code is the way it is. Each entry is a real bug or decision from building the prototype. **The "Invariant" lines are things not to break** — if a change moves them, it's a regression. `scripts/test.mjs` guards the big ones; the subtle matching ones need manual care.

## Parsing

### Position‑independent parsing
CAMS and KFintech differ, and PDF text extractors fragment rows and wrap ISINs onto the next line. Every field (NAV, market value, closing units, cost, ISIN) is parsed by its **own pattern**, tolerant of fields being **merged on one line or split across lines, in any order**. Wrapped ISINs are handled by looking ahead. **Never** parse by fixed column/line offsets.
> **Invariant:** all statement fixtures parse to their exact totals (see `handoff.md §2`).

### KFintech: ISIN torn mid‑character across a row wrap (2026‑07‑03)
First fixture with confirmed `Registrar : KFINTECH` folios (`vandana_kfintech.txt` — AXIS, Quant MF, Quantum Mutual Fund, UTI Mutual Fund) surfaced a new fragmentation shape: pdf.js's row‑grouping (group by rounded y, sort by x) sometimes packs the scheme‑header row so tightly against the adjacent "Registrar : KFINTECH/CAMS" box that the **ISIN itself is torn mid‑character** across two rows — not moved as a whole token (which the existing `_awaitISIN` lookahead already handled), but genuinely split, e.g. `...ISIN: INF109K01` / `V83(Advisor: DIRECT)`. Worse, the row‑grouping is inconsistent about where the "Registrar : ..." text lands: sometimes it trails the partial ISIN on the *same* row, sometimes it's squeezed onto its *own* row wedged between the partial ISIN and its continuation, e.g. `...ISIN: INF109K Registrar : CAMS` / `016E5(Advisor: DIRECT)`.
**Fix:** `stitchSplitIsins(lines)` — a pre‑pass before the main parse loop. Any `INF`‑prefixed run shorter than the required 9 trailing chars is unambiguously a truncated ISIN (nothing else in a CAS starts with `INF`), so it looks ahead up to 2 lines for the first one starting with enough `[A-Z0-9]` characters to complete it, splices the full ISIN back into the header line, and trims the consumed characters off the continuation line. Fixed in `portfolio-dashboard.html` first (source of truth), `reference/engine.js` regenerated from it per the mirror rule, and the equivalent ported into `app/src/parsing/cas/scheme.ts` (production port).
> **Invariant:** `vandana_kfintech.txt`'s 29 schemes all resolve a non‑empty ISIN (see `fixtures.spec.ts` "stitches an ISIN torn mid‑character..."), and its total matches ₹3,42,02,380 to the rupee.

### CAGR uses *remaining* cost basis
Early on, switched‑out funds showed phantom losses because CAGR used original cost. `weightedCAGR` uses the **remaining** cost basis (`costOverride`).
> **Invariant:** no phantom losses on funds that were partially switched/redeemed.

### All‑time return = portfolio XIRR (money‑weighted)
A naive point‑to‑point CAGR since inception was rejected as misleading for a portfolio with ongoing contributions. The headline "all‑time return" is the **money‑weighted XIRR** over every dated flow (incl. redemptions), so it truly reflects since‑inception performance.

### Opening‑balance & missing‑field handling
Funds with only an opening balance (no cost basis) show `hasCostBasis:false` → "—" rather than a wrong number. AXIS statement introduced the `"Valuation on"` MV label and a missing‑TCV fallback to invested. Treasury/savings funds infer as **debt**.

## Live‑NAV matching (the source of most bugs — read carefully)

The core tension: match each held fund to today's NAV **without ever applying the wrong fund's NAV**. The final design has four layers, all guarded by a plausibility gate.

### 1. ISIN is primary — with O↔0 folding
ISIN is stable across renames, so it's the primary key. **But registrar statements and AMFI sometimes disagree on letter‑O vs digit‑0 inside an ISIN** (e.g. `INF109K016O4`). `isin0()` folds `O→0`; AMFI ISINs are indexed under **both** spellings and lookups try both.
> **Invariant:** a statement ISIN with a letter O still matches AMFI's zero‑spelled ISIN (and vice‑versa).

### 2. Alias/plan‑aware name matching (renames)
Funds get renamed ("ICICI Prudential **Equity** Arbitrage" → "ICICI Prudential Arbitrage", w.e.f. 27‑May‑2026). Matching by name uses `liveKey = canonCore | planKey`, where `canonCore` resolves `NAME_ALIAS_GROUPS` **and** generically collapses `equity arbitrage → arbitrage` (a sweeping SEBI‑era rename class). `planKey` keeps Direct/Regular · Growth/IDCW distinct so a rename never matches the wrong plan.
> **Invariant:** old and new names of the same scheme resolve to the same `liveKey`; Direct/Growth never matches Regular/IDCW.
> **Gotcha:** a broad alias `['icici prudential multi asset','icici prudential dynamic']` was **removed** — "dynamic" is too generic and risked matching other funds. Keep aliases precise.

### Old statements and fund renames (2026‑07‑05, found via a real user statement)
A real CAS dated **January 2022** — tested for the first time end-to-end via the password-unlock feature — reported "1 of 4 holdings passed" with the Data Sources reason `"No matching live NAV found (no ISIN in statement, name match failed)"` for the other 3. One of them was **Axis Long Term Equity Fund**, which had no ISIN in this statement's format (matches the already-known ISIN-less-AXIS pattern from the `axis.txt` fixture) *and* had been renamed **Axis ELSS Tax Saver Fund** on 08‑Dec‑2023 (an industry-wide SEBI/AMC-driven ELSS renaming, confirmed via valueresearchonline.com) — so name-matching against today's live listings failed too, on a name that hadn't existed for years by the time this statement was tested.
**Fix:** added `['axis elss tax saver', 'axis long term equity']` to `NAME_ALIAS_GROUPS`. This works because `mfapiByName` already builds its search query from `canonCore(name)` (see `marketdata/sources/mfapi.ts`) — the alias makes an old-name statement search under the fund's *current* name, which is what's actually listed live.
> **Process note (standing instruction from the user):** when a fund fails to resolve a live NAV with reason "no ISIN in statement, name match failed" (or similar) on an **old** statement, check **morningstar.in** and **valueresearchonline.com** for a name change, merger, or wind-up before assuming it's an extraction bug — an old statement holding a since-renamed fund is a normal, expected failure mode this app's alias system exists to handle, not evidence of broken parsing. Add the confirmed old→new name pair to `reference/aliases.ts` (new name first, matching this file's existing convention) and a `liveKey` regression test in `harmonise.spec.ts`.
> **Open question, not resolved:** the "inconsistent ISIN presence" extraction-quality check (U4, `engine/extractionQuality.ts`) did not show the Convert/Instructions buttons on this upload. Two explanations are consistent with what's known and neither has been confirmed: (a) all 4 holdings in this statement are uniformly ISIN-less (in which case the check correctly stayed quiet, matching the already-established `axis.txt` precedent — a uniform format choice, not a defect), or (b) it *was* a genuine mix and something in the check's wiring didn't fire when it should have. Don't assume either answer without checking — if this recurs, ask whether ISIN is present for the *passing* fund specifically before concluding anything.

**Round 2 (same statement, same day):** after the Axis fix, the remaining 2 failures (both the *same* fund held across two folios — see the duplicate-key bug above) were **ICICI Prudential Equity Arbitrage Fund - Direct Plan - Growth (Advisor: DIRECT)**. This one wasn't a rename at all — CAMS had appended a `(Advisor: DIRECT)` broker/advisor-code annotation onto the raw scheme name (the value here happens to be the literal word "DIRECT" rather than a numeric broker code, which is what made it easy to mistake for a plan-arm token). `normName` only ever special-cased two specific parenthetical patterns (`(formerly...)` and `(non-demat)`) before falling through to a generic `[^a-z0-9 ]` strip that deletes the parens/colon but **keeps the words inside them** as loose tokens. So `coreTokens` produced `"icici prudential arbitrage advisor"` instead of `"icici prudential arbitrage"` — and `canonCore`'s alias lookup (`NAME_ALIAS_GROUPS`, which already had the correct `['icici prudential arbitrage', 'icici prudential equity arbitrage']` entry from a prior fix) is an **exact-string match**, so the stray "advisor" token silently broke it even though the right alias already existed.
**Fix:** added `.replace(/\(advisor:?[^)]*\)/g, ' ')` to `normName` (`engine/harmonise.ts`), stripping the whole annotation wholesale — same pattern as the existing `(formerly...)`/`(non-demat)` handling. Regression test in `harmonise.spec.ts` asserts `canonCore` on the real annotated string still lands on `"icici prudential arbitrage"`.
> **Lesson:** a missing alias isn't the only way name-matching fails on a real statement — a raw name carrying broker/account metadata in parens can silently pollute the *generic* normalization step even when the right alias is already in place. If a fund fails to match despite an existing alias, check `coreTokens(name)` for stray tokens before assuming the alias table is the problem.

**Round 3 (2026-07-06):** "Axis Arbitrage Fund - Direct Growth" also failed to resolve — a genuinely different, two-part bug in `marketdata/sources/mfapi.ts`'s `mfapiByName`, unrelated to aliases or `normName`:
1. **Search-query crowding.** mfapi.in's `/search` endpoint caps results at 15 with no relevance ranking (confirmed live: querying `axis` alone and `axis arbitrage` both return exactly 15). The search query was built from the bare `canonCore(name)` output ("axis arbitrage" — `canonCore`/`normName` deliberately strip "fund" for matching-invariance), and Axis has since launched an "Income Plus Arbitrage Active/Passive FOF" family with 15 IDCW/growth share classes that entirely fill the cap — "Axis Arbitrage Fund" itself never appears in the results. **Fix:** append `' fund'` back onto the search query only (not into `canonCore`/matching) — every real scheme name contains "fund", so this only narrows the search, never breaks a correct match.
2. **Scoring threshold structurally rejected short-core funds.** Independent of (1): the `want` token set used for candidate scoring was built from `canon + planKey` (e.g. "axis arbitrage direct growth" — 4 tokens), but a candidate's `toks` (from its own `canonCore`) never includes plan words at all (plan matching is already strictly enforced earlier via the `wantDirect`/`wantIdcw` `continue` filters). This guaranteed a fixed `want.size - toks.size == 2` gap for every genuinely-matching candidate, making the `Math.abs(...) * 0.1` penalty always exactly `0.2` — so the `bestScore >= 2` bar silently required **3+ core-token overlap**, meaning any fund with a canonical core of 2 words or fewer (e.g. "axis arbitrage") could never match regardless of query quality. **Fix:** build `want` from core tokens only (`canon.split(' ')`, no plan words), since plan matching is already handled separately.
> **Lesson:** both bugs were invisible in every existing test because every prior fixture (Kotak Small Cap, Axis ELSS Tax Saver, etc.) happened to have a 3-word-or-longer canonical core. A short, generic core (AMC name + one category word) is a normal, common shape for real fund names — not an edge case.

**Round 4 (2026-07-10):** a user-uploaded statement reported the same failure shape for **"Quantum Equity Fund Of Funds - Direct Plan Growth"** — no ISIN in the statement, name match failed across AMFI/captnemo/mfapi. Confirmed directly against mfapi.in and AMFI's live NAVAll.txt (ISIN INF082J01093, unchanged): the fund was renamed **"Quantum Diversified Equity All Cap Active FOF - Direct Plan Growth Option"**. Unlike the Axis ELSS case (a same-length industry-wide renaming), this rename added enough new descriptive words ("Diversified", "All Cap", "Active") that `coreTokens` produced two token sets sharing only "quantum"/"equity" — a Jaccard-ish overlap around 0.29, well under `fuzzyLive`'s 0.7 threshold, so even the fuzzy last-resort tier couldn't bridge it.
**Fix:** added `['quantum diversified equity all cap active fof', 'quantum equity of funds']` to `NAME_ALIAS_GROUPS`. Verified two ways, not just the alias-collapse unit check: (1) `canonCore`/`liveKey` on both names now match (`harmonise.spec.ts`); (2) an end-to-end scratch run through the real `resolveLiveNavs` pipeline, replicating the user's exact no-ISIN scheme, actually resolved — `{ matched: true, nav: 85.83, source: "AMFI" }` — confirming AMFI's real listed name text (fetched live, not assumed) canonicalizes to the same alias.
> **Reusable diagnostic recipe for this failure shape:** (1) `curl` mfapi.in's `/mf/search?q=<fund words>` to find the fund's current listing and ISIN; (2) `curl -L` AMFI's `NAVAll.txt` and grep the ISIN to get AMFI's exact name text (mfapi and AMFI don't always agree on formatting — verify both, don't assume); (3) run the OLD and NEW names through `coreTokens`/`canonCore` (a throwaway Vitest `it` block, not hand-derivation — the regex has enough special-casing that eyeballing it is unreliable) to get the exact alias-group strings; (4) add the alias, plus a `liveKey`-equality regression test; (5) re-verify end-to-end through the real `resolveLiveNavs`, not just the string-matching unit alone, since the plausibility gate and AMFI-vs-captnemo-vs-mfapi tier selection are separate code paths the alias fix doesn't touch but should still be confirmed compatible with.

**Round 5 (2026-07-11), same fund, two more bugs — and Round 4's own "end-to-end verification" turned out to be a false positive.** After the Round 4 alias fix, the user reported the *same two funds* now failing with a **different** message: `"Live market-data sources were unreachable (network or CORS block)"`. Round 4's step-5 verification had run `resolveLiveNavs` end-to-end and seen `source: "AMFI"` — but that scratch run was **Node/vitest**, where `fetch` doesn't enforce CORS. Direct-testing the same call in the *actual browser* this time (`javascript_tool`, not a Node script) showed AMFI's direct fetch failing with a real CORS error, exactly as `docs/DECISIONS.md`'s own N1/N3 notes already documented — Round 4's "confirmed end-to-end" was real in Node and never true in the environment the user actually reports bugs from. Lesson repeats the Nifty-benchmark incident two rounds earlier in this file: a Node-side check that's blind to browser networking semantics is not the same claim as "this works," and re-trusting an already-fooled verification method doesn't catch it — reproduce the failure *in the browser itself* before believing a fix.

Two real, independent bugs turned up once actually testing in-browser:
1. **`mfapiByName`'s query-narrowing heuristic breaks "FOF"-named schemes.** The "+fund" suffix (added for a prior Axis-crowding bug — see `docs/DECISIONS.md`'s Round 3, `marketdata/sources/mfapi.ts`) assumed every real scheme name literally contains the word "Fund." "Quantum Diversified Equity All Cap Active **FOF**" doesn't — confirmed directly via `curl`: the bare query returns exactly the right 4 matches, the "+fund"-suffixed query returns **zero**. **Fix:** try the bare query first; only fall back to the "+fund"-narrowed query if the bare one doesn't produce a confident match — covers both failure modes (crowded family *and* FOF-named) instead of picking whichever one the last bug happened to need. `searchAndScore` split out of `mfapiByName` so both query strings share the same scoring logic.
2. **The gap-rescue tier fires all mfapi.in lookups in parallel, and mfapi.in cannot handle that.** Measured directly in the browser: a *single* mfapi.in request is fast (~300ms-1s once a connection is warm), but firing 2 concurrent requests took **17.2 seconds** combined — mfapi.in appears to serialize or throttle concurrent requests from the same client, the same category of bug already found and fixed for the Nifty-50 benchmark fetch (`marketdata/sources/benchmark.ts`, two rounds earlier in this file) but not, at the time, generalized to this other mfapi.in call site. `resolve.ts`'s `Promise.all(gaps.map(s => mfapiByName(s.name)...))` guaranteed every unmatched fund's lookup fired at once, and 2+ concurrent lookups routinely exceeded the per-request timeout. **Fix:** changed the gap-rescue loop to sequential (`for` loop with `await`, not `Promise.all`) — the first lookup in a session may still be slow (a real, unavoidable cold-connection cost, observed up to ~17s), but it no longer competes with every other unmatched fund's lookup for the same contended connection; each one lands or fails on its own. Also raised `mfapi.ts`'s two `fetchJSON` timeouts from 9s to 20s, since 15s (this round's first attempt) still weren't quite enough margin against the worst observed cold-start case.
> **Process lesson, stated plainly:** this file now has *three* incidents (Nifty benchmark, this fund's Round 4, this fund's Round 5) where a fix looked correct because it was checked in Node or checked once and happened to land in a fast window, and *all three* were actually intermittent browser-networking failures. If a bug report mentions "CORS," "unreachable," or "network," the verification has to happen with `javascript_tool` calling the real function in the real browser tab — not `vitest`, not a mocked test, and not a single lucky run. Time the call with `performance.now()` and re-run it 2-3 times before believing "fixed."

### Same scheme across multiple folios (2026-07-06)
A statement can list the same scheme (same ISIN) under two different folios — e.g. two SIPs set up at different times. Before this fix, `Scheme`/`Fund` was strictly one-per-(folio, scheme) pair from the parser all the way to the UI (`engine/portfolio.ts`'s `funds` was a straight `schemes.map(analyzeScheme)`, no grouping step anywhere). This was already known to cause a React duplicate-key warning (fixed earlier by keying on `${isin||name}-${folio}`), but the underlying **data** still showed the scheme twice everywhere — Fund Cards, Data Sources, Full Holdings, the allocation drill-down — each with its own partial (wrong) avgCost/XIRR/CAGR computed only from that one folio's transactions. Confirmed against a real fixture (`vandana_kfintech.txt` has 3 ISINs each duplicated across folios) and a real user statement.
**Why this can't be fixed by merging two already-computed `Fund` objects:** XIRR and CAGR are not additive or averageable — `analyzeScheme` derives them from that scheme's own cash-flow/lot list (`gains.ts`'s per-lot STCG/LTCG apportionment, `xirr.ts`'s flow-based solve). Correct consolidation requires merging the **raw transactions** of both folios into one list, then running the existing per-scheme analysis once — not touching two already-derived numbers after the fact.
**Fix:** `groupSchemesByIdentity(schemes)` (`engine/scheme.ts`) groups by `isin || name` and, for a collision, concatenates+date-sorts `txns`, sums `closingUnits`/`marketValue`/`costValue` (NaN-safe — only sums when at least one side is finite, so "neither folio ever reported a cost value" doesn't become a false `0`), unions `folio` into a joined string (`"F1, F2"`), and keeps the more recent `nav`/`navDate`. Wired into `analyzePortfolioFromSchemes` as the very first step, before `liveFor`/`valDate`/the `funds` map — so every downstream consumer (funds[], houses, allocation, portfolio XIRR, the chart series) automatically sees one row per scheme. `analyzeScheme` itself is unchanged; it just now always receives an already-merged `Scheme`.
> **Invariant:** `pf.funds` never contains two entries with the same `isin || name`, regardless of how many folios the source statement split a scheme across. Portfolio-level totals (`totalValue`, `portXirr`, series) are unaffected by consolidation — they were always computed by summing over every scheme's real transactions regardless of how many `Scheme` records those transactions were split across; only the **per-scheme** figures (avgCost, XIRR, CAGR, gains split) were wrong before, since each folio's partial view of the transaction history produced a different, individually-incorrect answer.

### 3. Fuzzy core‑token fallback
If ISIN and exact name both miss, `fuzzyLive` matches within the **same plan** by core‑token Jaccard ≥ 0.7. Catches name variations without matching across plans.

### 4. Plausibility gate — the safety net that fixed the big bug
**Symptom:** uploading `alok_2026`, the correct ₹2,51,65,629 became ₹1,97,01,983 because ICICI Arbitrage and Multi‑Asset got **wrong** live NAVs (matched to a low‑NAV fund). **Fix:** `navPlausible(nav, stmtNav)` — since statement and live dates are days apart, reject any live NAV outside **⅓×–3×** of the statement NAV. `liveFor` returns only plausible matches; an implausible/missing match is treated as a **gap** and **rescued** via mfapi.in → Yahoo (also plausibility‑gated), rather than silently corrupting the valuation or being dropped.
> **Invariant:** a live NAV wildly off the statement NAV is never applied; the fund is rescued or kept at statement NAV (and flagged by Data Check). Legitimate same‑week NAVs are **not** rejected.

> **Two layers, not one (discovered while writing the port's tests, 2026‑07‑03):** `navPlausible` is checked in *two* places — `analyzePortfolio`'s own `liveFor()` filters every candidate before it ever reaches `analyzeScheme`, **and** `analyzeScheme` has its own independent `liveRejected`/`rejectedNav` check on whatever `live` it's handed. In practice, calling through `analyzePortfolio` means `analyzeScheme`'s check is unreachable dead code — `liveFor` already stripped anything implausible. It only fires for a caller that invokes `analyzeScheme` directly with an unfiltered candidate (defense in depth, and how `app/src/engine/scheme.spec.ts` tests it). **If you're writing a regression test for "wrong-fund rejection" via `analyzePortfolio(text, {live})`, assert `navLive:false` and the protected total — don't expect `liveRejected:true` there; test that flag against `analyzeScheme` directly instead.** (Same structure in both `portfolio-dashboard.html` and the `app/` port — not a porting divergence.)

### `liveSource` display string was accumulating per-fund, not per-source (2026‑07‑03)
**Symptom, caught by actually running the `app/` port in a browser** (not by any unit test — nothing was asserting the *string content* of `pf.liveSource`): the valuation banner read "Valued at live NAVs … via mf.captnemo.in + mf.captnemo.in + mf.captnemo.in + mf.captnemo.in …", once per fund captnemo resolved. The old mfapi.in branch was worse — it didn't even append, it **hardcoded** `source = source ? 'AMFI + mfapi.in' : 'mfapi.in'`, silently overwriting whatever had actually matched (e.g. erasing "mf.captnemo.in" and claiming "AMFI + mfapi.in" even when AMFI never resolved anything). Present in both `portfolio-dashboard.html`'s `fetchLiveNavs` and the `app/` port's `resolve.ts` — a straight, faithful port carries a source's real bugs forward too.
**Fix:** stop mutating `source` inside each per-fund match loop. Build it **once**, at the end, from the `diag` flags that were already being tracked correctly (`amfiOk`/`captnemoUsed`/`mfapiUsed`/`yahooUsed`), in a fixed order, joined by `' + '`.
> **Invariant:** `pf.liveSource` names each contributing source **at most once**, regardless of how many funds it resolved.
> **Lesson:** this class of bug — cosmetic, in a display string built from mutated state across a loop — is exactly what unit tests miss (the *numbers* were always right) and a five-minute live-browser look catches immediately. Don't skip the manual pass just because the automated suite is green.

### Doubled ₹ symbol in three places (2026‑07‑03)
`inr()` already prepends `₹`, but three call sites additionally wrote a literal `₹` before `${inr(...)}`, producing "₹₹43,24,678": `chartInvestedValue`'s caption (both `c.value` and `c.invested`), and the commentary's "Today your ₹… is roughly X% equity" line. Same bugs in both the prototype and the port — `inr(...)` alone is correct wherever a formatted rupee amount is interpolated; only use the literal `₹` when the interpolated call is `.toLocaleString(...)` directly (which doesn't add the symbol itself — see `sourceFor`'s "rejected NAV" message for a correct example of that pattern). Caught only by rendering the page and reading the actual text, not by any test — the *numeric* values were always correct, only the extra literal `₹` was wrong.

### AMFI parsing robustness
`parseAmfi` parses each row **from the right** (date/nav last; scheme name may contain `;`), scans **every** ISIN token on the row (growth/idcw columns vary), and indexes raw + `isin0`. This fixed "Kotak `INF174K01211` not getting NAV" and similar.

### Fetch reliability (CORS)
AMFI `NAVAll.txt` has **no CORS headers** → needs a proxy, and single proxies are flaky. Fix: **race all proxies in parallel** (`Promise.any`) — codetabs (reliable), allorigins raw, corsproxy, thingproxy, + allorigins‑JSON — take the first **complete** response (`looksLikeAmfi` rejects truncated/garbage by requiring many ISIN tokens). mfapi.in is CORS‑native and used for rescue by canonical name; it and Yahoo are per‑fund fallbacks.
> Note for production: prefer CORS‑native, ISIN‑capable sources (`mfapi.in`, `mf.captnemo.in`) + your own cached AMFI edge function over public proxies. See `plan.md`.

### `mf.captnemo.in` added as a CORS‑native ISIN primary (2026‑07‑03)
Per `plan.md`'s NAV‑hardening recommendation, added `captnemoByIsin(isin)` (`GET mf.captnemo.in/nav/:isin`, `Access-Control-Allow-Origin: *`, no logs). It runs **in parallel** with the AMFI proxy race (not after it) for every held scheme with an ISIN, since it needs no proxy at all — the AMFI race is the part of the pipeline actually vulnerable to network/proxy blocking. Verified live: in one test run every AMFI proxy failed (one 413, one DNS failure, others silently blocked) and captnemo alone still resolved 7/7 held holdings, including `INF109K016O4` (ICICI Prudential Arbitrage) — the exact fund/ISIN from the plausibility‑gate bug above — correctly plausibility‑gated and displaying the AMC's current post‑rename name.
> **Gotcha:** captnemo does **not** fold O↔0 itself — a raw O‑spelled ISIN it doesn't recognise 404s as `{"error":"Invalid ISIN"}` rather than falling back internally. `captnemoByIsin` tries the raw ISIN then `isin0(isin)` explicitly, same pattern as the AMFI `byIsin` lookup.
> **Precedence:** when both AMFI and captnemo resolve the same ISIN, captnemo's result wins (assigned after AMFI's `Object.assign`) — both are ISIN‑exact so this is about resilience/freshness, not trust; either passes the same `navPlausible` gate.
> mfapi.in has no ISIN‑search endpoint (confirmed against its docs) — it stays a name‑only rescue after captnemo/AMFI leave a gap.

## Portfolio views & charts

### Exited/zero‑balance folios: hidden but retained
Fully‑redeemed folios are **hidden** from the holdings/AMC/sources views (cleaner), but their flows and realised gains are **retained** in the portfolio realised total and the since‑inception money‑weighted return. AXIS is the proof case — its all‑time return comes entirely from an exited round‑trip.
> **Invariant:** hiding exited folios must not change the all‑time return or realised gains.

### Charts value curve ties out to MV
No historical NAVs exist, so `buildPortfolioSeries` reconstructs value by interpolating each fund's price between its transaction points and today's NAV, tracking units. Units are **anchored to the statement closing balance** so the curve's endpoint equals the headline market value **exactly**. Drawdown uses a **TWR** index (contributions removed) — raw value can't show drawdown because contributions push it up.
> **Invariant:** `series.line` last value == `PF.totalValue`.

### Rolling returns: first‑year solution
Trailing‑1Y needs 12 months of history. For the first year it shows the **expanding since‑inception** (annualisable) return, drawn distinctly, then switches to true rolling 1Y. AXIS (short history) correctly shows only the expanding segment.

### Commentary must differ by horizon
`commentaryBand(age, z)` produces materially different guidance for a 30‑yr‑old (long horizon → 85–100% equity, human‑capital argument) vs a 50‑yr‑old (→ 75–90%, begin glide/ballast), grounded in Bogle's "age in bonds" + 25–75% guardrails and the S‑shaped glide path (de‑risk mostly 45–60). Keep the "educational, not advice" disclaimer.

### Geography is an estimate
`geoFor` uses `FUND_META.geo` where set, else India for domestic sleeves + a "Gold & Commodities" bucket + keyword inference. The CAS has no country data — label it as approximate.

## Input / MarkItDown

### One normaliser, identical results
`normalizeInput` makes the parser **source‑agnostic**: it flattens Markdown tables (drop `|---|` rules, `|`→space) and strips MarkItDown's spliced rotated CAMS page‑footer (**forward and character‑reversed** — MarkItDown sometimes reverses rotated text). It's idempotent on plain pdf.js text.
> **Invariant:** MarkItDown Markdown and pdf.js text of the **same** statement parse to the **same** total (guarded by test `[2]`).

### Why a local bridge for "Convert PDF to Markdown"
A browser page can't run Python MarkItDown. `markitdown_server.py` is a tiny localhost‑only (127.0.0.1:8765), no‑storage bridge the button POSTs to. In Claude Desktop, the `markitdown` MCP tool can do the conversion and the Markdown is dropped/pasted in instead.

### pdf.js stays the default; MarkItDown is a conditional fallback, not "primary" (2026‑07‑05)
`plan.md` originally said "MarkItDown primary, pdf.js fallback" — the shipped app never followed that (pdf.js parses immediately on drop; MarkItDown needed an extra click **and** a locally‑running Python bridge). A review confirmed there's no LLM/token cost in this pipeline either way — both paths are 100% local — so the real trade‑off is parsing *robustness* (MarkItDown reconstructs CAS tables more cleanly) against *zero‑setup* (pdf.js needs nothing extra). Direct instruction locked pdf.js as the permanent default (tasks.md U4): MarkItDown now surfaces only as a conditional fallback (a Convert + Instructions button pair) when `engine/extractionQuality.ts` detects a genuine problem, never as a parallel default choice.

**`assessExtractionQuality` deliberately does NOT reuse `engine/datacheck.ts`'s `reconciles` check.** `reconciles` sums the same `f.marketValue` values `Portfolio.totalValue` itself sums (`engine/portfolio.ts`, `totalValue = funds.reduce((a,f)=>a+f.marketValue,0)`) — it's nearly tautological and would almost never actually fire from a fragmented PDF extraction. Don't reach for it here again without re-deriving why it doesn't work for this purpose.

**A "missing folio" rule was tried and dropped** after testing against the 5 real golden fixtures, not just synthetic ones: 3 of them (alok_2025/2026's Parag Parikh Liquid Fund, vandana_kfintech's UTI Nifty 50 Index Fund) have a scheme whose folio genuinely isn't captured — a real, pre‑existing, harmless registrar‑format‑variance gap with zero effect on those fixtures' exact golden totals. A missing folio doesn't correlate with actual bad data here; shipping that rule would only produce a wrong, distracting nudge on statements the app already handles correctly. Always check a new extraction-quality rule against the real fixtures before shipping it, not just hand‑built corrupted cases.

### Password-protected statements (2026‑07‑05, tasks.md U5)
CAMS/KFintech CAS PDFs are almost always password-protected. `ingest/router.ts`'s `IngestSource` type and `ingest/pdf.ts`'s `pdfToText(data, password)` already carried an optional password parameter before this — unused, since nothing detected the need or prompted for one. Confirmed by reading `pdfjs-dist`'s own source that with no `onPassword` callback registered (this app's usage), `getDocument(...).promise` **rejects** with a `PasswordException` rather than hanging — a plain catch‑and‑retry works, no need for the interactive `onPassword` API.

**MarkItDown's `convert()` cannot take a password — checked its actual source, not assumed.** It accepts `**kwargs` but never forwards them to `pdfplumber`/`pdfminer`; a `password=` kwarg is silently dropped. `markitdown_server.py` works around this by decrypting the PDF itself with `pypdf` *before* handing it to MarkItDown (`PdfReader.is_encrypted` to detect; `reader.decrypt(password)` returns `PasswordType.NOT_DECRYPTED` — falsy — on a wrong password, without raising) — a standard decrypt‑then‑process pattern, not a fork of MarkItDown. `pypdf` is imported lazily, only after a first `_md.convert()` attempt fails and the file looks encrypted, so an unencrypted-PDF user never needs it installed.

> **Known gap, logged rather than silently skipped:** the `markitdown_server.py` changes were written and carefully proofread but **never executed** — no Python interpreter is available in the dev sandbox this was built in (only Windows Store app-execution-alias stubs). Similarly, `ingest/pdf.spec.ts` tests the `pdfToText` password-error translation against a **mocked** `pdfjs-dist` (no real encrypted PDF fixture — none could be generated without Python/`pypdf`). Both should get a real end-to-end pass (a genuine encrypted CAS PDF, the bridge actually running) before this is fully trusted.

### The password block's width must never depend on its flex-row siblings (2026‑07‑05)
The first version of the password prompt lived inside `.drop-col`, a `flex:1` item sharing a row (`.wrap`) with `.upload-status` (whose message text changes length across the upload flow) and the Refresh/Convert/Instructions/Clear-Data buttons. Flex items default to `min-width: auto`, so `.upload-status`'s text won't shrink below its own natural width — meaning `.drop-col`'s actual rendered width was always just *whatever space was left over* after the other items claimed theirs, a width that shifts every time the status message's length changes. The password block's `margin: auto` centering inside that column only produced a *visible* gap once the column happened to be wider than the block's own max-width — so the same block could render flush-left at one status-message length and visibly re-center at another, which is exactly what was reported ("initially left-aligned... then changes after a moment to fully centred"). Confirmed the mechanism with a temporary debug hook (`App.tsx`, `window.__debugPw`, removed after) driving the real `pendingPassword` state while measuring `getBoundingClientRect()` over time and across viewport widths.
> **Invariant:** the password block (and anything like it) must size itself independently of its neighbors' content — it now sits as a sibling of `.wrap` with the exact same box recipe (`max-width: 1080px; margin: 0 auto; padding: 0 28px`), so its width can never be a function of what else is happening in the upload bar's button row. If a future change moves it back inside that row "to save space," this bug will return.

### Fund list keys must include folio, not just ISIN/name (2026‑07‑05 — found via a real user's real statement)
The password-unlock feature's first real-world use (a genuine password-protected CAMS PDF, not a synthetic test) surfaced a "dashboard fails to refresh and update" report. The browser console showed the decrypt-and-parse actually **succeeded** — the real bug was downstream: repeated React "Encountered two children with the same key" warnings for one specific fund. The statement held that scheme across **two folios** (a legitimate, ordinary real-world shape — e.g. two SIPs opened at different times into the same fund), and `FundCards.tsx`, `TopHoldings.tsx`, and `DataSources.tsx` all keyed their per-fund list items by `f.isin || f.name` alone — identical for both folio entries, so React saw duplicate keys and could reuse/confuse DOM nodes between them on re-render, which is what "stuck, not updating" actually looks like from the outside.
**Fixed** by keying all three lists by `` `${f.isin || f.name}-${f.folio}` `` instead — folio is per-scheme-block and distinguishes exactly this case.
> **Testing note:** this class of bug is invisible to `renderToStaticMarkup` — confirmed empirically that it does **not** run React's key-uniqueness validation at all (zero `console.error` calls even with genuinely duplicate keys), which is why none of this project's existing render-smoke specs caught it. The regression test (`FundCards.spec.tsx`) instead uses a real client-side render (`react-dom/client`'s `createRoot` + `act`, both already available from the existing `react`/`react-dom` deps — no new dependency) to exercise the actual reconciliation path where the warning fires. Verified the test actually catches the bug by temporarily reverting the fix and confirming it fails, then restoring it.
> **Broader lesson:** the synthetic test PDFs used throughout this feature's development never exercised a fund held across multiple folios — a real, ordinary statement shape a same-fixture set can miss entirely. Prefer testing shapes drawn from what real statements actually look like, not just what's easy to construct synthetically, especially for anything list/identity-related.

### Dark/Light theme toggle: "Terminal Deck" vs "The Ledger" (2026-07-08)
Two full themes, not one: a dark amber-phosphor CRT-terminal look ("Terminal Deck" — JetBrains Mono, scanline texture) and a warm cream editorial-ledger look ("The Ledger" — Fraunces headlines, Courier Prime typewriter data, a red ledger-book margin rule), toggled instantly via a button (crescent moon / sun icon, labeled with the mode a click switches **to**, not the current one — "Dark Mode" while on The Ledger, "Light Mode" while on Terminal Deck). Two mockups were shown for review before implementation; the user asked for both, togglable, rather than picking one.

**Mechanism:** a single `data-theme="dark" | "light"` attribute on `<html>`, with every color/font token defined twice in `tokens.css` (`:root, [data-theme='dark']` and `[data-theme='light']`) under the *same* variable names. Toggling writes one DOM attribute; the browser's native CSS cascade repaints every `var(--token)` consumer in one style-recalc pass — no React re-render of the dashboard tree, no runtime style computation. `ThemeToggle.tsx` owns only its own label/icon state (persisted to `localStorage`, applied before first paint in `main.tsx` since `<body>` has no content besides `#root` — nothing to flash).

**Fonts are self-hosted, not loaded from Google Fonts.** `index.html`'s CSP is `font-src 'self'`, `script-src 'self'` — a deliberate choice for an app that processes real, unencrypted personal financial statements client-side. `@fontsource/{jetbrains-mono,fraunces,courier-prime}` bundle the actual `.woff2` files as same-origin Vite assets, so both themes' fonts load with zero external network requests and no CSP change. Because `@font-face` only downloads a font lazily on first use, `main.tsx` eagerly calls `document.fonts.load(...)` for all three families at startup (fire-and-forget) — otherwise the *first* toggle would visibly wait on a font fetch before the new typeface painted; every toggle after that was already instant. (An `index.html` `<link rel="preload">` was considered and rejected: Vite hashes these files under `dist/assets/` at build time, so a static path in `index.html` can't reliably reference them.)

**Making the retheme actually total, not partial:** `tokens.css` was already the project's single source of truth in principle, but `deck.css` had accumulated ~23 hardcoded hex values (mostly literal duplicates of the *old* single dark theme's token values, hand-copied instead of `var()`-referenced) plus several `rgba(...)` tint backgrounds in `app.css` computed from those same old values — both invisible to the toggle. All were converted to `var(--token)` (tint backgrounds via `color-mix(in srgb, var(--token) N%, transparent)`, which re-derives from whichever theme is active rather than needing a token per tint). Two TS files hold color logic that can't live in CSS (`categories.ts`, `geography.ts`); these were changed to hold the literal string `'var(--cat-equity)'` etc. rather than hex — React inline styles (and, confirmed empirically, raw SVG `fill="..."` attributes) resolve `var()` natively, so no theme-awareness was needed in the TS layer at all.
> **Preserved a subtlety while doing this:** `.legrow:hover`/`.legrow.active` in `app.css` relied on a `.deck { --card: #252a33 }` local override (removed as part of the same hardcode-cleanup) to be visibly lighter than the tile background it sits on. Simply deleting that override without replacement would have made hover invisible (background == tile background). Added a proper `--card-hover` token instead, used for that plus `.deck-btn:hover`/`.deck-advt:hover`, which had been using a *border* color (`--frame-line`) for what is semantically a hover *background*.
> **Text-on-accent-fill:** places like `.dc-icon`/`.empty-state-icon`/`.gbtn:hover` need a color that's legible against a bright/saturated accent fill (`--brass`, `--green`, `--pos`) in *both* themes. `var(--paper)` (not `var(--ink)`, and not a hardcoded white) is the right choice: accent colors are, by design, always the opposite brightness of the page background in either theme — near-black `--paper` on Terminal Deck's bright amber/cyan fills, cream `--paper` on The Ledger's dark forest-green/oxblood fills.
> **Testing note:** `ThemeToggle.spec.tsx` uses a real client render (`react-dom/client` + `act`, the same technique established for the folio duplicate-key regression) since the toggle's entire effect is a DOM-attribute side effect invisible to `renderToStaticMarkup`. Verified as a real guard: broke the click handler, confirmed the click-cycle test failed, restored it.

### Terminal Deck contrast pass: the yellow tint was the *structural* colors, not the accent (2026-07-08)
First feedback on Terminal Deck after using it for real: "doesn't look professional... yellow tint across the dashboard." The root cause wasn't the accent color — it was that every *structural* token (`--ink`, `--paper`, `--card`, `--frame`, `--line`, `--muted`) had been given the same warm amber bias as the accent, in the name of a cohesive "amber-phosphor" identity. Since `--ink` is the primary text color, used on nearly every label and number on the page, a saturated amber (`#E8C477`) as the *default text color* reads as an overall yellow wash, not a deliberate accent — and since `--paper`/`--frame`/`--card` were all close in both lightness *and* hue, nested surfaces barely differentiated from each other (low contrast).
**Fix:** structural colors became a genuinely neutral gray ramp with clearly distinct lightness steps (`--paper: #0A0B0D` → `--frame: #101216` → `--card: #1A1D22` → `--card-hover: #262A31`, `--ink: #EDEFF2`, `--muted: #8A8F99`, `--line: #333842`) — amber survives only as an accent (`--green`/`--brass`/`--cat-equity`), used sparingly, never as body text. The scanline/vignette atmosphere overlay (`body::before` in `tokens.css`) was also re-tinted from amber (`rgba(255,179,0,...)`) to neutral white (`rgba(255,255,255,...)`), since a page-wide amber wash on top of already-amber text compounded the problem. The Ledger palette was untouched — the user confirmed it separately, and none of its tokens had this issue (its accent, oxblood/forest-green, was never doubling as the body text color).
> **Lesson:** a themed accent color and the theme's neutral/structural palette are different design decisions — reusing one hue for both is what "on-brand but unreadable" usually turns out to be. A "yellow tint" complaint about an amber-accented theme is almost always about where the amber leaked into text/surface roles, not about the accent itself needing to change.

Requested as a Framer Motion effect — scale 1.05 + a softened, more diffuse box-shadow on hover, eased with a spring (`type: 'spring', stiffness: 300, damping: 20, mass: 0.6`) so it settles with a slight overshoot rather than linearly — applied consistently to every genuinely "boxed" surface across the dashboard, in both themes.

### Nifty 50 benchmark: a short-history fund must not fake a long window (2026-07-09)
User-reported: the "Nifty 50" comparison figures under XIRR "looked wrong." Root cause, confirmed by querying mfapi.in directly: IDBI's long-running Nifty 50 tracker fund (the one this benchmark used to resolve to) **stopped reporting NAVs in mid-2023** — 2+ years stale, correctly excluded by the existing `isRecent`/`STALE_DAYS` check in `fetchNiftyBenchmark`. The only other fund matching the strict name filter and still reporting is Angel One's Nifty 50 Index Fund, which **launched 2025-05-28** — about 14 months of history. `benchmarkCagr`'s "nearest point" lookup had no bounds-check: asked for a CAGR from an 11-year-old portfolio inception date, it silently used the fund's own first-ever NAV (the *nearest* point it had) as a stand-in for "11 years ago," computing that fund's real ~1.1-year return and presenting it as if it covered the full 11-year "All-Time" window.
**Fix:** `benchmarkCagr` now requires both `nearest(from)` and `nearest(to)` to fall within `COVERAGE_TOLERANCE_MS` (30 days) of the actually-requested dates — generous enough for ordinary weekend/holiday nearest-match slop in a daily NAV series, nowhere near enough to accept a gap of years. Outside that tolerance, it returns `null` rather than a real-looking wrong number; `KpiRail.tsx`'s existing `niftyAllTime != null` check already renders `—` in that case, so no UI change was needed beyond the fix at the source. The trailing-1-year window is unaffected (Angel One's fund did exist a year ago), and its computed figure (~−5.1%) was cross-checked directly against the fund's real mfapi.in NAV history for 09-Jul-2026, landing close to the user's independently-sourced reference figure (~−4.87%) — the small remaining gap is ordinary fund tracking-error/expense-ratio drag versus the raw index, not a bug.
> **Lesson:** a "nearest point in a series" lookup needs a coverage bound whenever the series can legitimately be shorter than the requested window — otherwise it silently substitutes "the closest thing I have" for "the thing you asked for," which reads as a real answer until someone checks the underlying dates. This is the same category of defensive check as the live-NAV plausibility gate (`navPlausible`) — reject and disclose rather than silently produce a wrong-but-plausible-looking number.

### Feedback system: a local companion server, same pattern as the MarkItDown bridge (2026-07-09)
A floating "Feedback" button (bottom-right-of-center-edge, matching the theme toggle's position/size per the entry above) opens a modal: category (`Bug Report` / `Feature Request` / `General Feedback`) + free-text message, posted to a small Express server the user runs themselves — `server/feedback-server.js`, listening only on `127.0.0.1:8766`. This mirrors `markitdown_server.py`'s established pattern exactly (a companion local process, not a hosted backend), for the same reason: this app processes real personal financial data and has a deliberately strict CSP (`connect-src 'self' ...`); a feedback backend is a new network destination, so it gets the same "you run it, it's local-only" treatment rather than an always-on remote service. `index.html`'s CSP `connect-src` was extended by exactly one origin, `http://127.0.0.1:8766` — same shape as the existing `127.0.0.1:8765` MarkItDown entry.
**Backend:** Express + `better-sqlite3` (a single-file local DB, no separate DB server needed — same "minimal setup" ethos) + `sanitize-html`. Validates `category` against an explicit allowlist and `message` for non-empty/under 5000 chars, then **strips all HTML markup** from the message before storing it (defense in depth: nothing renders this as HTML today, but a plain-text-only guarantee at the point of storage means a future admin/display surface can't be XSS'd by old data). Uses parameterized `better-sqlite3` queries throughout (no string-built SQL) to rule out injection. CORS restricted to localhost/127.0.0.1 origins only.
**Actually run and verified, not left as an unexecuted stub** (unlike `markitdown_server.py`, which has no Python interpreter available in this sandbox) — Node/npm *are* available here. Installed real dependencies (including building `better-sqlite3`'s native module via `npm approve-scripts` + `npm rebuild`), ran the server, and drove a real end-to-end submission from the actual browser preview: confirmed the CORS preflight (`OPTIONS` → 204), the POST (→ 201), the UI's "Thanks — your feedback has been sent" confirmation, and the row landing in `feedback.db` with the message correctly sanitized — not just unit-tested in isolation.
**Frontend error handling:** a `fetch` failure (server not running) surfaces a specific, actionable message — "Couldn't reach the feedback server at 127.0.0.1:8766. Start it with `cd server && npm install && npm start`" — the same "tell the user exactly what to run" convention as the MarkItDown bridge's unreachable-bridge message in `App.tsx`. Server-side validation errors (bad category, empty message) surface the server's own error text directly.
> **Testing:** `Feedback.spec.tsx` uses the same real-client-render technique as `ThemeToggle.spec.tsx` (`react-dom/client` + `act`) since opening the modal, submitting, and the fetch-driven state transitions are all side effects invisible to `renderToStaticMarkup`. The backend has its own `node --test` suite (`server/feedback-server.test.js`) — a temp SQLite file per test, real HTTP requests via `fetch` against an ephemeral-port `express` instance, not mocked.

### Production feedback: a Pages Function forwarding to a webhook, not the local server (2026-07-12, task D3)
The local companion server above only exists on *your own* machine — once the app is deployed, a real visitor's `127.0.0.1:8766` is their own laptop, which never has this server running, so every production submission would silently fail. Fixed with the same shape as N2/D1's AMFI edge function: `app/src/server/feedbackWebhook.ts` exports a plain Web-standard `(Request, webhookUrl) => Promise<Response>` handler (no platform imports), wrapped by a thin `app/functions/api/feedback.ts` Pages Function entry point. It re-validates everything the client already validates (never trust client input server-side) — allowed categories, non-empty message, under 5000 chars — then forwards `{category, message}` to a webhook URL read from `env.FEEDBACK_WEBHOOK_URL`, a Cloudflare **secret**, never a public `VITE_` var and never committed.
**Payload shape, chosen for zero-config compatibility:** `{ text, content, category, message, submittedAt }` — `text` is Slack's Incoming Webhook field, `content` is Discord's; sending both means the exact same deployed code works with either provider (or a generic Zapier/Make/n8n catcher, via the raw fields) with no code change, only which URL you paste into `FEEDBACK_WEBHOOK_URL`.
**Mention-injection is a real risk here that the local server never had:** this endpoint accepts free text from anyone on the public internet and forwards it into a Slack/Discord channel — a message like `@everyone` or `<!channel>` would ping the entire channel from a stranger's "bug report." `neutralizeMentions()` inserts a zero-width space after every `@` before forwarding (breaks both platforms' mention syntax; the `@` itself stays readable, so a genuine email address or handle in feedback text isn't mangled beyond recognition). Deliberately *not* reusing the local server's `sanitize-html` (full HTML-stripping) here — this handler only ever puts the message into a JSON string field sent to a webhook, never renders it as HTML anywhere in this pipeline, so the actual injection surface is mention syntax, not markup; adding an HTML sanitizer would be unused weight on the edge bundle for a risk that doesn't exist in this specific path.
**`Feedback.tsx`** now resolves its endpoint as `VITE_FEEDBACK_URL || 'http://127.0.0.1:8766/api/feedback'` — unset (the default, unchanged from before D3), local dev behaves exactly as it always has; set to `/api/feedback` (same-origin, in the Cloudflare dashboard per `docs/DEPLOY.md`), it hits the new Function instead. The "server not running, start it with `cd server && npm start`" error message only makes sense in local dev, so it's now conditional on which endpoint is active — the edge path shows a generic "couldn't send feedback, try again" instead.
**Verified end-to-end without touching any real external service:** built the app with `VITE_FEEDBACK_URL=/api/feedback` baked in, ran `wrangler pages dev` with `FEEDBACK_WEBHOOK_URL` in a local (gitignored) `.dev.vars` pointed at a throwaway local HTTP listener on the same machine, then drove a real submission through the actual Feedback form in a live browser — confirmed the `201`, the "Thanks — your feedback has been sent" UI state, and the listener receiving the exact expected payload with `@everyone` correctly neutralized to `@​everyone`. 10 new unit tests (`feedbackWebhook.spec.ts`, mocked `fetch`, same convention as `amfiNav.spec.ts`) cover validation, the not-configured-yet 500, successful forwarding, mention neutralization, and webhook-unreachable/non-OK handling.

### Nifty 50 "All-Time" benchmark was blank for the Sample Portfolio — two real bugs, not one (2026-07-10)
The Sample Portfolio's inception is 2018-09-07 (~7.8 years). `fetchNiftyBenchmark` (`marketdata/sources/benchmark.ts`) only ever searched mfapi.in for `'Nifty 50 Index Fund'` — a single query. Confirmed directly against the live API: that query returns exactly 15 hits, capped, sorted alphabetically by AMC, entirely filled by Angel One/IDBI/Motilal Oswal's several variants before the alphabet ever reaches UTI. UTI's direct-growth Nifty 50 tracker — real, still-reporting, continuous NAV history back to 2013-01-02 — was never excluded by the existing `isRecent`/coverage-tolerance filters; it was simply never returned by the search at all. Of the two funds that single query *could* find, neither covers a 7.8-year window: IDBI's goes back to 2013 but stopped reporting in 2023 (correctly rejected as stale); Angel One's only launched 2025-05-28.

**First fix attempt (insufficient on its own):** broadened `NIFTY_QUERIES` to six parallel searches (generic + one per major AMC), merged and deduped by scheme code. This found UTI's fund correctly *in Node* (`vitest`, no browser connection semantics) — but retesting live in the actual browser via `javascript_tool` (`await import(...).then(m => m.fetchNiftyBenchmark())`, timed with `performance.now()`) showed it intermittently returning `null` after ~9.7s, right at the 9s per-request timeout. A single isolated browser request measured 277ms; the same six queries fired via `Promise.all` measured ~7-10s *each*. Root cause: a request to a cold `api.mfapi.in` connection can take 8-11s in this environment, and firing multiple queries at once means every one of them pays that cold-start cost independently — none can reuse a connection the others haven't finished opening yet. More candidates found by a wider query set doesn't help if the fetch that finds them times out.

**Actual fix:** trimmed `NIFTY_QUERIES` down to the two that are ever productive — the original generic query plus `'UTI Nifty 50 Index Fund'` (Nippon India's history duplicates UTI's 2013 start, adding a request for no new coverage; HDFC/ICICI Prudential/SBI's schemes use `"Cumulative"`/`"IDCW"` naming instead of `"Growth"` and never pass the growth-option filter regardless of how they're searched for) — and switched from `Promise.all` to a sequential `for` loop over those two queries, so only the *first* request can ever hit the cold-start cost; the second reuses its now-warm connection. Per-request timeouts raised from 9s/12s to 15s to give the observed worst-case (measured once at 10.66s) real headroom. Verified directly in the browser three separate times after this change (fresh reload each time, `performance.now()`-timed): consistently `found: true, count: 3326, first: 2013-01-02, last: today` — and end-to-end in the live dashboard, "Nifty 50: 10.9%" (All-Time) and "Nifty 50: −3.6%" (1Y) both render after the normal statement→live-NAV loading transition, no console errors, across three fresh page loads.

**Process note:** the first fix passed its own unit tests and a Node-side live-data check — both of which are blind to browser connection-pooling/cold-start behavior, since Node's `fetch` has no equivalent semantics and the mocked tests don't model timing at all. Confirming "the underlying data problem is solved" is not the same as confirming "this works in the environment the user actually reported the bug from" — the second pass only found the real cause by reproducing the failure *in the browser itself* (`javascript_tool` calling the real function directly, bypassing React) rather than re-trusting the same Node-side check that had already given a false pass once.

### 100-fund coverage sweep: legacy share-class crowding, then a confident-wrong-match regression (2026-07-11)
Built a permanent 100-fund test fixture (`app/tests/fixtures/coverage_sample.txt` / `coverage_sample_no_isin.txt`) spanning SEBI's four top-level scheme groups (Equity/Debt/Hybrid/Other, 25 funds each), sourced from real AUM-ranking research and cross-verified against live mfapi.in/AMFI data — the first of a 5-part testing plan for "a new statement uploads a fund that doesn't get recognized." The ISIN-based path passed 100/100 immediately (the common case is robust). The no-ISIN variant, forcing 100% name-based resolution, found two real bugs in `mfapiByName` (`marketdata/sources/mfapi.ts`):

**Bug 1 — legacy pre-Direct-plan share classes crowd out the target.** AMCs with a long history (ICICI Prudential, Aditya Birla Sun Life) have dozens of pre-2013 Institutional/Retail/IDCW share classes that predate Direct plans existing at all, sharing the same base name as the fund being searched for. mfapi.in's `/search` caps at 15 results with no relevance ranking, so both the bare query and the existing "+fund" narrowed query (see the Axis Arbitrage entry above) could return 15 results that are *entirely* these legacy variants — the real Direct-Plan-Growth scheme never appears in either result set, since all of them already contain the word "fund" too. Fixed by adding a third query tier appending the plan word itself (`+direct`/`+regular`) — confirmed live that this narrows "ICICI Prudential Liquid Fund" from 15 crowding results to 9, all Direct-plan.

**Bug 2 — the tier-3 fix above introduced a confident-but-wrong match.** For "Aditya Birla Sun Life Liquid Fund," the bare query's 15-result cap didn't just exclude the real fund — it included an unrelated *sibling* fund ("...Gilt Plus - Liquid Plan - Growth - Direct Plan") that happened to be the only Direct/non-IDCW candidate present. Under the old scoring (`overlap - |sizeDiff| * 0.1`), that sibling's two extra tokens ("gilt", "plus") cost only 0.2, so it still cleared the confidence threshold and `mfapiByName` returned it immediately — a *wrong* match, silently, which is worse than the old *no* match, since a later, untried query (the "+fund" tier) actually contained the correct fund cleanly.

Root cause: the matcher stopped at the **first** query tier that cleared a low, mostly-overlap-based confidence bar, rather than considering all tiers and picking the best. Fixed with two changes: (1) `searchAndScore`'s formula changed to `overlap * 2 - extra` (extra tokens penalized at full weight instead of 0.1, and the threshold raised from 2 to 4 proportionally) — an unrelated-but-overlapping product family now scores well below a true exact match; (2) `mfapiByName` now runs all three query tiers and keeps the single best-scoring match across them, short-circuiting early only when a tier produces an *exact* match (`extra === 0`, nothing missing) — since nothing later could ever outscore that, preserving the fast path for the common case while fixing the ambiguous one.

Verified live in the browser (not just Node/vitest, per the standing rule below): all three originally-failing funds now resolve to their correct schemes and NAVs. Re-ran the full 100-fund no-ISIN sweep twice — 99/100 and 100/100 (the one transient miss retried clean in isolation in 1.1s, confirming network flakiness during the long sequential run, not a real match gap).

### Retiring the HTML/mirror duplication (2026‑07‑03)
The engine existed in three places at once: the single‑file `portfolio-dashboard.html` prototype (2,579 lines), `reference/engine.js` (a regenerated mirror of its inlined `<script>`, 783 lines), and the ported `app/src/**` (3,927 lines). This wasn't just untidy — it caused the **same bug to be fixed three times** in one session (the `liveSource` accumulation bug and the doubled‑₹ caption bug below both needed a fix in the HTML, a mirror regeneration, and a separate fix in `app/`). A critical review concluded the HTML's "living spec" value was spent once `app/` had verified fixture parity, and the ongoing sync tax outweighed any benefit. `portfolio-dashboard.html`, `reference/engine.js`, and `scripts/_sync_engine.mjs` were deleted; `app/` is now the only place the engine exists.
> **Invariant:** there is exactly one implementation. If you ever find yourself reconciling two copies of the same logic again, that's a regression of this decision — stop and consolidate.
